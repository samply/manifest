The Samply manifest


Who is the Samply community?


The Samply community is a group of software developers and scientists from research institutions at different locations, founded to build software for scientific purposes, networking and federated research. As an open source community, it is our goal to provide the developed Samply software from individual projects to all members of the community.

Originally established in the German Cancer Consortium, the Samply initiative's open-source licensed software is used by several European scientific projects and research networks.

Communication and organisation of the community:


All Members of the Samply community are expected to comply with the principles and conventions of the Samply manifest.

Every member of the Samply community adheres to the rules of the Berlin Code of Conduct.

The Samply community is self-organized and self-commited to divide the workload, to implement the requirements of the software and to solve pull requests.

We have an open communication culture. Although we develop and use Samply in a variety of independent research projects, we commit to communicating ideas, problems and changes openly.

For discussions, questions, complaints or feature requests you can contact the developers via Samply mailing list, Issue Tracker and Samply chat.

Our working language for documentation is English. In meetings we will choose our working language based on the preferences of the attendees.

Members of the Samply community with long-term activity in developing Samply software may become members of the Samply committee, either by invitation from the current committee or by asking for admission.

The members of the Samply committee decide by majority about organisational topics like changes of internal workflows. Modifications to the Samply manifest must be decided unanimously by the committee members.

Each Samply component is curated by one or more maintainers. Those who engage intensely in the development and enhancements of a component are predestined to become its maintainer (see list of component maintainers).

The Samply committee determines the component maintainers in agreement with potential candidates. Component maintainers automatically become members of the Samply committee.

Meetings should take place biannually and every member of the community is invited to participate. In addition, the Samply committee plans web conferences to inform the members about news, changes and the roadmap.

The appointment and location of a meeting is organised by the committee. The meeting's agenda will be prepared by all participating members.

If you would like to join the Samply community, please register to our Samply mailing list by sending an e-mail to join(at)samply.de. You can leave the community the same way. Communication on the mailing list is archived and might be made public.

Principles and guidelines of software development:


Our source code is licensed under AGPLv3 and is provided "as is" without warranty on our public GitHub repositories.

Every member of the Samply community works to the principles of agile software development.

It is assumed that each member of the Samply community contributes to the processing of requirements and pull requests according to their own abilities and interests.

We aim to provide Docker buildfiles for each Samply application, and to publish them at Docker Hub.

We do not have Samply-wide release trains. However, each project employing Samply can maintain 'Samply distributions' to establish their own release trains.

Component maintainers are the contact person for the component and allowed to approval pull requests and release new versions. They are responsible to fix detected bugs and keep the component compatible with linked components.

The members of the Samply community are self-obligated to adhere to our criteria catalog for pull requests.


Our criteria catalogue for pull requests:


Make your code as clean as possible according to coding best practices (check out 'Clean Code' by Robert C. Martin for reference).

Projects will be version-controlled according to Gitflow. There are two permanent branches, 'master' and 'develop'. Additional to that, temporary branches for features, hotfixes and release can be established.

The software versioning uses three-figure version number, follows the guidelines of Semantic Versioning 2.0.0.

The master branch is the current stable release version. The develop branch is a copy of the latest master branch which contains additional new developed features and changes. Its the base for the next release.

Never work in the master branch! Establish a new temporary branch from the current develop branch for development (e.g. feature, hotfix), which will be merged on the develop branch after successful pull request.

It is possible to add the appendix 'SNAPSHOT' for versions under development. Avoid dependencies from a SNAPSHOT version, except the concerned version is also a SNAPSHOT. The develop and master branch must never depend on SNAPSHOT versions.

Pull requests should be addressed by Samply committee members and especially the responsible component maintainers. To solve a pull request the acceptance of a component maintainer is required.

Component maintainers need to be aware that pull requests are of high priority and should be tended to within 2 days. Delays in pull request approval might otherwise slow down the entire developement process.

Only the component maintainers are allowed to merge branches into develop and to push from develop into the master branch for a new release. New features should be merged from develop into master branch within two weeks.

For Java software we support one OpenJDK-version on release. The Samply committee discuss and determine the supported version and potential version-upgrades.

We keep external dependencies as low as possible. External libraries have to be suggested to the Samply committee and agreed by the affected component maintainers before use.

Each modification should be communicated to the component maintainer. Also other involved community members should be informed promptly.

Whenever a modification on interfaces is done, we will do a system test.

We use a Samply license header for our software. Embedding this header is a requirement for each Samply component.

For each release we make sure to keep the compatibility between connected Samply components (for combinations check the releases-tabs or the docker-compose-files of the deployments at GitHub/Samply). If necessary, we will modify affected Samply components to ensure compatibility.

For very good reasons, it is possible to break the rules of this criteria catalogue. Please inform affected community members and the Samply committee about a rule break promptly.
